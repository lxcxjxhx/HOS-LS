# HOS-LS 漏洞严重等级评估标准

## 等级定义

### CRITICAL (严重)
**定义**：漏洞可被直接利用，导致系统完全沦陷或核心数据大规模泄露。

**典型场景**：
- 远程代码执行（RCE）
- 严重身份认证绕过（如 JWT 密钥硬编码）
- 核心加密算法被破解
- 数据库完全暴露

### HIGH (高危)
**定义**：漏洞可被利用获取敏感数据或提升权限，对业务安全产生重大影响。

**典型场景**：
- SQL 注入（可联合查询）
- 未授权访问核心业务功能
- 敏感数据泄露（密码、密钥、令牌）
- **Swagger/API 文档未授权暴露（导致攻击者收集更多信息）**
- 垂直越权
- 敏感文件泄露（.env, .git 等）

### MEDIUM (中危)
**定义**：漏洞可被利用获取有限信息或执行有限操作，需要特定条件配合。

**典型场景**：
- XSS（存储型或反射型需用户交互）
- CSRF（需要用户操作）
- **Swagger 端点暴露无认证（信息泄露，攻击者可借此发现更多漏洞）**
- 不安全依赖（已知漏洞）
- CORS 配置错误
- 会话管理问题

### LOW (低危)
**定义**：漏洞影响有限，或需要极其复杂的条件才能利用。

**典型场景**：
- XSS（仅 DOM-based，作用域有限）
- 日志敏感信息暴露
- 配置建议问题
- 点击劫持（需用户交互）

### INFO (信息)
**定义**：不是漏洞但值得关注的发现，或需要进一步确认的可疑模式。

**典型场景**：
- 代码风格问题
- informational findings
- 需要更多上下文才能确认的可疑模式
- 潜在问题但无明确利用路径

## 漏洞类型与建议等级对照表

| 漏洞类型 | 建议等级 | 说明 |
|---------|---------|------|
| SQL 注入 | HIGH/MEDIUM | 取决于是否可联合查询、堆查询等 |
| XSS 存储型 | HIGH | 可窃取会话、劫持用户 |
| XSS 反射型 | MEDIUM | 需用户点击特定链接 |
| 命令注入 | CRITICAL/HIGH | 直接执行系统命令 |
| CSRF | MEDIUM | 需用户操作配合 |
| SSRF | HIGH/MEDIUM | 取决于可访问的内网资源 |
| XXE | HIGH/CRITICAL | 取决于是否可读取本地文件 |
| 路径遍历 | MEDIUM/HIGH | 取决于可访问的文件范围 |
| 未授权访问 | HIGH/MEDIUM | 取决于可访问的功能范围 |
| 敏感数据泄露 | HIGH/MEDIUM | 取决于数据类型 |
| **Swagger 暴露** | **MEDIUM** | **信息泄露，帮助攻击者发现更多漏洞** |
| 硬编码凭证 | HIGH | 密钥、密码等 |
| 弱加密 | MEDIUM/HIGH | 取决于算法强度和用途 |
| CORS 错误配置 | MEDIUM | 取决于允许的域 |
| JWT 算法 none | CRITICAL | 直接认证绕过 |
| OAuth2 配置错误 | MEDIUM/HIGH | 取决于影响的授权类型 |

## AISVS 验证等级定义

AISVS（AI Security Verification Standard）定义了三个验证级别，覆盖不同安全保障需求：

### Level 1 - 基础级
**定义**：适用于 AI 系统的基本安全控制基线，针对通用 AI 应用场景的最低安全要求。

**覆盖章节示例**：
- C2.1.1 - 输入验证与清理
- C7.1.1 - 日志记录要求
- C9.1.1 - 基础访问控制

**适用场景**：
- 内部工具 AI 系统
- 非生产环境部署
- 低风险数据处理场景

### Level 2 - 标准级
**定义**：适用于生产环境 AI 系统，要求完整的安全控制措施。

**覆盖章节示例**：
- C2.2.1 - 高级输入验证
- C3.1.1 - 模型访问控制
- C7.2.1 - 完整日志与审计
- C10.1.1 - 数据加密要求

**适用场景**：
- 生产环境 AI 系统
- 处理敏感业务数据
- 面向用户的 AI 服务

### Level 3 - 增强级
**定义**：适用于高价值或高风险 AI 系统，要求最严格的安全控制。

**覆盖章节示例**：
- C2.3.1 - 深度输入验证与净化
- C3.3.1 - 强化访问控制与多因素认证
- C7.3.1 - 实时威胁检测与响应
- C11.1.1 - 对抗性攻击防护
- C12.1.1 - 模型完整性验证

**适用场景**：
- 金融、医疗等高风险 AI 应用
- 处理关键基础设施决策
- 高价值数据处理的 AI 系统

## AISVS 需求编号格式

AISVS 标准采用统一的需求编号格式，便于追溯和引用：

**格式规范**：`v{version}-C{chapter}.{section}.{requirement}`

**组成部分**：
| 组成部分 | 含义 | 示例 |
|---------|------|------|
| v{version} | 标准版本号 | v1.0, v2.0 |
| C{chapter} | 章节编号 | C2, C7, C11 |
| {section} | 章节内分组 | .1, .2, .3 |
| {requirement} | 具体需求编号 | .1, .2, .3 |

**编号示例**：
- `v1.0-C2.1.1` - 版本 1.0，第二章第 1 节第 1 条需求
- `v1.0-C7.2.3` - 版本 1.0，第七章第 2 节第 3 条需求
- `v2.0-C11.3.2` - 版本 2.0，第十一章第 3 节第 2 条需求

**章节映射**：
- C2 - 输入安全与验证
- C3 - 访问控制与身份认证
- C7 - 日志与监控
- C9 - 数据保护
- C11 - 对抗性攻击防护
- C12 - 模型安全

## CVSS 评分参考

### CVSS v4.0 评分范围

| 等级 | 分数范围 | 描述 |
|------|---------|------|
| CRITICAL | 9.0-10.0 | 漏洞可被直接利用，导致系统完全沦陷 |
| HIGH | 7.0-8.9 | 漏洞可被利用获取敏感数据或提升权限 |
| MEDIUM | 4.0-6.9 | 漏洞可被利用获取有限信息或执行有限操作 |
| LOW | 0.1-3.9 | 漏洞影响有限，或需要极其复杂的条件才能利用 |
| INFO | 0.0 | 不是漏洞但值得关注 |

### CVSS v4.0 评分因素

CVSS v4.0 在 v3.1 基础上增加了更多评分维度：

**基础评分因素**：
- **攻击向量** (AV): Network / Adjacent / Local / Physical
- **攻击复杂度** (AC): Low / High
- **所需权限** (PR): None / Low / High
- **用户交互** (UI): None / Required
- **影响范围** (S): Changed / Unchanged
- **机密性影响** (C): High / Low / None
- **完整性影响** (I): High / Low / None
- **可用性影响** (A): High / Low / None

**新增评分因素（v4.0）**：
- **攻击复杂度-其他** (AC-P): Low / High - 量化其他攻击复杂度因素
- **可用性-其他** (A-P): High / Low / None - 量化对其他组件的可用性影响
- **保密性-其他** (C-P): High / Low / None - 量化对其他组件的机密性影响
- **完整性-其他** (I-P): High / Low / None - 量化对其他组件的完整性影响
- **防攻击利用** (PA): Not Defined / Not Applicable / Attack with Prevention / Attack with Mitigation / Attack with Restriction / Attack with Regular Enforcement / Attack with Prevention and Detection
- **安全目标强度** (CR): Not Defined / High / Medium / Low
- **恢复目标强度** (RR): Not Defined / High / Medium / Low
- **价值目标密度** (TD): Not Defined / High / Medium / Low / None

## AI 安全专项严重性评估

AI 系统存在传统应用之外的安全风险，需专项评估：

### 提示注入（Prompt Injection）

| 类型 | 建议等级 | 说明 |
|------|---------|------|
| 直接提示注入 | CRITICAL/HIGH | 攻击者直接控制模型输出 |
| 间接提示注入 | HIGH/MEDIUM | 通过外部输入注入恶意指令 |
| 上下文操纵 | MEDIUM | 影响模型决策但不直接执行恶意指令 |

**典型场景**：
- 用户输入包含隐藏指令导致模型执行非预期操作 → CRITICAL
- 外部数据源被污染导致输出异常 → HIGH
- 通过大量正常请求绕过检测 → MEDIUM

### 模型输出安全

| 类型 | 建议等级 | 说明 |
|------|---------|------|
| 敏感数据泄露 | HIGH/CRITICAL | 模型输出密钥、密码、个人信息 |
| 有害内容生成 | HIGH | 模型生成违法或有害内容 |
| 幻觉信息 | MEDIUM/LOW | 模型生成虚假但看似可信的信息 |
| 版权内容泄露 | MEDIUM | 模型输出受版权保护的内容 |

**典型场景**：
- 对话机器人输出后端数据库结构 → CRITICAL
- AI 助手泄露用户会话 Token → HIGH
- 生成虚假医疗建议 → HIGH
- 生成看似合理但不准确的引用 → MEDIUM

### 编码攻击（Encoding Attack）

| 类型 | 建议等级 | 说明 |
|------|---------|------|
| 多语言编码绕过 | HIGH | 通过特殊编码绕过输入过滤 |
| Unicode 混淆 | MEDIUM/HIGH | 利用 Unicode 特性进行混淆 |
| 嵌套编码 | MEDIUM | 多层编码增加检测难度 |
| 协议层攻击 | HIGH/CRITICAL | 利用协议转换特性进行攻击 |

**典型场景**：
- SQL 注入通过 URL 编码绕过检测 → HIGH
- 命令注入通过 Unicode 同源字符混淆 → HIGH
- XSS 通过混合编码绕过 WAF → MEDIUM
- 利用 AI 系统的编码解析差异 → CRITICAL

### 对抗性攻击

| 类型 | 建议等级 | 说明 |
|------|---------|------|
| 对抗样本攻击 | HIGH | 通过扰动输入导致模型误判 |
| 数据投毒 | CRITICAL/HIGH | 污染训练数据影响模型行为 |
| 模型提取 | HIGH/MEDIUM | 通过查询窃取模型知识 |
| 后门攻击 | CRITICAL/HIGH | 在模型中植入隐蔽后门 |

## 特殊场景处理

### Swagger/API 文档暴露
- **无认证的 Swagger UI** → 至少 MEDIUM
  - 原因：攻击者可通过 Swagger 发现更多 API 端点和漏洞
- **有认证但信息详细** → LOW
  - 原因：正常用户可见，但攻击者难以利用

### 信息泄露类
- 泄露测试数据 → LOW
- 泄露生产敏感数据 → HIGH
- 泄露可利用的配置信息 → MEDIUM/HIGH

### 业务逻辑漏洞
- 需根据实际影响评估
- 同类型漏洞在不同业务中可能有不同等级

## 综合评估原则

在同时存在多个评估标准时，按以下优先级确定最终等级：

1. **AISVS Level** 与 CVSS 评分冲突时，以实际影响范围为准
2. **AI 安全专项漏洞** 优先使用 AI 安全专项评估标准
3. **复合漏洞** 取各部分漏洞等级中的最高值
4. **利用条件** 影响最终等级的重要因素
5. **业务上下文** 需纳入最终评估考量
